Пишем proto-файл

Вы пришли на этот курс изучать C++, но пришлось разбираться в JSON, XML, потом в CMake, а теперь ещё и в Protobuf. Количество разнообразных языков растёт как на дрожжах. Но это нормально, такова судьба программиста. Одного языка редко бывает достаточно.

Если вы дошли до этого момента курса, значит, уже неплохо разбираетесь в C++, и усвоить несложный синтаксис proto-файлов не составит труда.

Подробное описание устройства файлов Protobuf есть в документации на сайте библиотеки. Мы разберём только основные возможности. Поддерживаются два различных синтаксиса: proto2 и proto3. Это напоминает ситуацию с языком Python, у которого версии Python 2 и Python 3 развиваются параллельно.

Мы будем использовать только синтаксис proto3 как более современный. Это нужно указать в первой строке proto-файла:

syntax = "proto3";

Будем создавать приложение — адресную книгу. Поэтому сохраним код в файл person.proto. Добавим сообщение с информацией о контакте:

syntax = "proto3";

message Person {
    string name = 1;
    int32 age = 2;
}

Описание сообщений похоже на описание класса или структуры в C++. Но есть несколько отличий:

  • сообщения не имеют методов,
  • в конце определения не ставится точка с запятой,
  • нужно указывать номера полей.

Зачем нужны номера полей? Выберите основное назначение.

  • Чтобы поля корректно сопоставлялись при изменениях.

  • Чтобы задать порядок загрузки и сохранения.

  • Чтобы ссылаться на поля по номерам, а не именам.

Изменения — это жизнь. Гарантировать, что основная структура данных не поменяется не может даже программа с очень давней историей. А в молодой программе данные, скорее всего, будут меняться, как сердце красавицы или ветер мая.

Поэтому основная польза, которую приносят номера полей — безопасные изменения. Можно менять порядок, переименовывать, добавлять новые и удалять старые поля сообщения. Возможность загрузить то, что было сохранено ранее, при этом не теряется. Главное, чтобы сохранялся номер поля. Он должен быть уникальным в пределах одного сообщения.

Простейший proto-файл готов. Скомпилируем его, передав на вход собранной ранее программе protoc:

<путь к пакету Protobuf>/bin/proto --cpp_out . person.proto

При генерации заданы следующие параметры:

  • --cpp_out . — указание, что нужно сгенерировать код на C++ и положить его в папку ., то есть в текущую папку.
  • person.proto — имя входного proto-файла.

Из 6 строк файла person.proto программа protoc создала более шестисот в файлах person.pb.h и person.pb.cc. Точное количество зависит от версии Protobuf. Расширение .h вам хорошо знакомо, а .cc — это альтернатива для .cpp. Оба файла начинаются с комментария:

// Generated by the protocol buffer compiler.  DO NOT EDIT!

Он говорит, что файл сгенерирован компилятором protocol buffer, и его не нужно редактировать. Последнее требование даже выделено прописными буквами.

Почему не нужно править файлы, сгенерированные Protobuf?

  • Это нарушает лицензию библиотеки.

  • Правки сотрутся после следующей генерации.

  • Вы можете нарушить правильную работу библиотеки.

Это правило вполне универсально: никогда не правьте файлы, сгенерированные автоматически. Всё равно ваши правки будут уничтожены при следующей перегенерации — это часть сборки программы. Даже если вы исключили процесс перегенерации из процедуры сборки, внесение изменений в исходник будет заблокировано.

Также сгенерированные файлы не нужно загружать в репозиторий. Важно, что в репозитории должно быть только то, что не получается автоматически из других файлов того же репозитория.

Продолжим изучать person.pb.h. Нас интересует класс Person, сгенерированный для сообщения Person.

В нём имеются:

  • конструктор копирования,
  • конструктор перемещения,
  • конструктор по умолчанию,
  • копирующее присваивание,
  • перемещающее присваивание,
  • специальные методы Protobuf,
  • методы взаимодействия с данными сообщения: name и age.

Разберём методы взаимодействия с данными сообщения name и age подробнее. Они идут в секции accessors. Для получения значений name и age есть методы, которые называются так же, как и само поле. Для изменения есть методы set_name и set_age.

Перед методами стоит комментарий, который показывает, к какому полю сообщения Person они относятся:

class Person ... {
...

  // string name = 1;
  void clear_name();
  const std::string& name() const;
  void set_name(const std::string& value);
  void set_name(std::string&& value);
  void set_name(const char* value);
  void set_name(const char* value, size_t size);
  std::string* mutable_name();
  std::string* release_name();
  void set_allocated_name(std::string* name);

...

  // int32 age = 2;
  void clear_age();
  ::PROTOBUF_NAMESPACE_ID::int32 age() const;
  void set_age(::PROTOBUF_NAMESPACE_ID::int32 value);

...
};

Protobuf создал и другие методы. Их рассмотрим подробнее в следующем уроке. А сейчас попробуем расширить сообщение, добавив в него новое поле — адрес, который сам будет отдельным сообщением:

syntax = "proto3";

message Address {
    string street = 1;
    uint32 building = 2;
}

message Person {
    string name = 1;
    int32 age = 2;
    Address address = 3;
}

Сообщение, подобно классу C++, определяет новый тип данных, который можно использовать для создания полей в других сообщениях.

Перегенерируем файлы person.pb.h и person.pb.cc, заново запустив программу protoc. Как и ожидалось, в proto.bp.h появился новый класс class Address с методами для работы с полями street и building. В классе Person добавились новые методы для работы с полем address:

// .Address address = 3;
bool has_address() const;
void clear_address();
const ::Address& address() const;
::Address* release_address();
::Address* mutable_address();
void set_allocated_address(::Address* address);

Продолжим расширять список контактов и добавим возможность хранить номера телефонов. Чтобы показать, что у одного контакта телефонов может быть несколько, добавим перед описанием поля слово repeated:

syntax = "proto3";

message Address {
    string street = 1;
    uint32 building = 2;
}

message Person {
    string name = 1;
    int32 age = 2;
    Address address = 3;
    repeated string phone_number = 4;
}

Слово repeated создаёт массив. Перегенерируем код и посмотрим, какие публичные методы появились у класса Person:

// repeated string phone_number = 4;
int phone_number_size() const;
void clear_phone_number();
const std::string& phone_number(int index) const;
std::string* mutable_phone_number(int index);
void set_phone_number(int index, const std::string& value);
void set_phone_number(int index, std::string&& value);
void set_phone_number(int index, const char* value);
void set_phone_number(int index, const char* value, size_t size);
std::string* add_phone_number();
void add_phone_number(const std::string& value);
void add_phone_number(std::string&& value);
void add_phone_number(const char* value);
void add_phone_number(const char* value, size_t size);
const ::PROTOBUF_NAMESPACE_ID::RepeatedPtrField<std::string>& phone_number() const;
::PROTOBUF_NAMESPACE_ID::RepeatedPtrField<std::string>* mutable_phone_number();

Это основные методы для работы с массивом, которые позволяют:

  • phone_number_size — узнавать количество телефонов,
  • phone_number — узнавать телефон по индексу,
  • set_phone_number — менять телефон по индексу,
  • add_phone_number — добавлять телефон в массив.
