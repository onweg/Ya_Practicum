Оптовая передача — наборы объектов

Вы изучили ситуацию, при которой в функцию требуется передать один объект. Но когда нужно передать целый набор объектов одного типа при неизвестном количестве, вариантов становится больше.

Передаём контейнер

Самый простой способ передать неизвестное заранее количество объектов в функцию — принять контейнер по ссылке:

void PreprocessFile(const path& file_name, const std::vector<path>& include_paths);

В чём недостаток передачи std::vector из этого примера? Считается, что функция не принимает владение вектором, ей не нужна копия. Выберите один или несколько вариантов.

  • Это неэффективно.

  • Нужно передавать по значению.

  • Неудобно и неэффективно использовать эту функцию, если объекты хранятся в контейнере другого типа, например, deque.

  • Неудобно и неэффективно использовать эту функцию, если требуется передать только часть объектов.

  • Из сигнатуры неясно, собирается ли функция менять объекты.

Часто использование такого способа допустимо. Например, если функция работает только с определённым контейнером и требует соблюдения его свойств. Тогда конвертации не избежать. В этом случае лучше пусть пользователь функции подстраивается под неё.

Другой пример — когда планируется использовать функцию в строго определённом контексте. Если во всей программе include-пути хранятся в векторе, то абсолютно нормально принимать их в виде константной ссылки на вектор.

Как обычно принимают набор объектов алгоритмы стандартной библиотеки?

  • Так же, контейнером.

  • Парой итераторов.

  • Каждый элемент — в отдельный аргумент.

Передаём полуинтервал

Алгоритмы стандартной библиотеки не принимают контейнер в качестве аргумента. Они используют более универсальный способ и принимают диапазон в виде пары итераторов:

template <typename RandomIt>
void std::sort(RandomIt first, RandomIt last);

Вы тоже можете выбрать подобный подход передачи диапазона:

template <typename InputIt>
void PrintBookings(InputIt first, InputIt last);

В чём недостатки этого подхода? Выберите один, или несколько вариантов.

  • Такая передача неэффективна.

  • Из сигнатуры неясно, какие типы допустимы.

  • Из сигнатуры неясно, собирается ли функция менять объекты.

  • Из сигнатуры неясно, что функция принимает диапазон, а не просто два элемента.

Отличие функции sort от PrintBookings в том, что последняя требует объекты строго определённого типа, в то время как sort может отсортировать практически любой диапазон. Из сигнатуры это неясно.

Некоторые алгоритмы, содержащие в своём названии _n, используют другой способ: они принимают итератор и количество.

template <typename InputIt, typename Size, typename OutputIt>
OutputIt std::copy_n(InputIt first, Size count, OutputIt result);

Передаём последовательность

Иногда функция работает с набором объектов, расположенных последовательно в памяти. Так делает метод ostream::write, о котором говорилось в предыдущем спринте:

ostream::write(const char* buffer, size_t buffer_size);

На самом деле ostream — это псевдоним для шаблонного класса basic_ostream, позволяющего работать с символами разных типов. Но здесь код упрощён: сигнатура метода приведена так, как если бы ostream был классом.

Похожим образом устроен один из конструкторов строки:

string::string(const char* c, size_t count);

И здесь сигнатура упрощена: string — это не класс, а всего лишь псевдоним для basic_string.

Для этих объектов указатель выполняет функцию итератора, поэтому такая передача похожа на алгоритмы с _n. Из сигнатуры может быть не очевидно, что указатель соотносится с числовым параметром. Например, указатель может передаваться для того, чтобы сделать аргумент необязательным, а число обозначать что-то другое.

Данные каких контейнеров могут быть представлены так же — указателем и числом? Выберите один или несколько вариантов.

  • std::vector

  • std::map

  • std::unordered_map

  • std::string

  • std::array

В C++20 для хранения указателя и количества введён специальный тип std::span. Он относится к vector или array так же, как string_view относится к string: предоставляет лёгкую оболочку, которая не хранит элементы. Отличие span от string_view в том, что если в нём неконстантный тип, то его элементы допускают изменение.

Как нужно передавать span в функцию?

  • По значению — объект лёгкий

  • По ссылке — объект тяжёлый.

  • По ссылке — копирование имеет побочный эффект.

  • По ссылке, если функция собирается менять элементы, на которые указывает span. По значению, если не собирается.

Перепишем функции потоков с использованием span:

#include <iostream>
#include <span>
#include <vector>

using namespace std;

istream& Read(istream& input, span<char> buffer) {
    input.read(buffer.data(), buffer.size());
    return input;
}

ostream& Write(ostream& out, std::span<const char> buffer) {
    out.write(buffer.data(), buffer.size());
    return out;
}

Можно использовать такую функцию, передавая в неё vector или array, поскольку эти типы неявно конвертируются в span:

vector<char> buffer(100);
Read(cin, buffer);

Либо вы можете создать span, явно передав указатель и размер:

if (cin.gcount() >= 1) {
    // выводим без первого символа
    Write(cout, {buffer.data() + 1, size_t(cin.gcount() - 1)});
}
