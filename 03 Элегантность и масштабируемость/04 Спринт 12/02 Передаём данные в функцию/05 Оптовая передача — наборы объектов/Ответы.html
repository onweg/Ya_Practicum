Ответы на задания

В чём недостаток передачи std::vector из этого примера? Считается, что функция не принимает владение вектором, ей не нужна копия. Выберите один или несколько вариантов.

  • (-) Это неэффективно.

Передача по ссылке будет занимать ничтожно мало времени по сравнению с обработкой всех объектов.

  • (-) Нужно передавать по значению.

Вектор — тяжёлый объект, его нужно передавать по ссылке.

  • (+) Неудобно и неэффективно использовать эту функцию, если объекты хранятся в контейнере другого типа, например, deque.

Если бы мы хранили пути в deque или list, то при вызове функции пришлось бы создавать вектор и копировать в него все элементы.

  • (+) Неудобно и неэффективно использовать эту функцию, если требуется передать только часть объектов.

В этом случае опять же пришлось бы создавать вектор и копировать элементы.

  • (-) Из сигнатуры неясно, собирается ли функция менять объекты.

Вектор принимается по константной ссылке, это говорит о том, что менять его элементы функция не будет.

Как обычно принимают набор объектов алгоритмы стандартной библиотеки?

  • (-) Так же, контейнером.

Некоторые алгоритмы реализованы в виде методов контейнеров, например sort у контейнера list, но отдельные алгоритмы контейнер не принимают.

  • (+) Парой итераторов.

Да, это более универсальный способ.

  • (-) Каждый элемент — в отдельный аргумент.

Количество передаваемых объектов может быть любым — на этапе написания сигнатуры функции оно не всегда известно.

В чём недостатки этого подхода? Выберите один, или несколько вариантов.

  • (-) Такая передача неэффективна.

Эффективность очень хорошая — передаются два лёгких итератора

  • (+) Из сигнатуры неясно, какие типы допустимы.

Об этом можно будет иногда догадаться из названия функции, но конкретный тип элементов не указывается явно.

  • (+) Из сигнатуры неясно, собирается ли функция менять объекты.

Информации о константности объектов в сигнатуре нет. Это существенный минус.

  • (-) Из сигнатуры неясно, что функция принимает диапазон, а не просто два элемента.

Передача пары итераторов — стандартный паттерн, который должен сразу наталкивать на мысль, что имеется в виду диапазон. Укрепит догадку название аргументов first и last. Если бы требовалось передать два объекта, передали бы сами объекты или ссылки, но не итераторы.

Данные каких контейнеров могут быть представлены так же — указателем и числом? Выберите один или несколько вариантов.

  • (+) std::vector

  • (-) std::map

  • (-) std::unordered_map

  • (+) std::string

  • (+) std::array

Как нужно передавать span в функцию?

  • (+) По значению — объект лёгкий

  • (-) По ссылке — объект тяжёлый.

Указатель и число — вовсе не тяжёлый.

  • (-) По ссылке — копирование имеет побочный эффект.

Указатель и число — это по сути два числа, копируются они без побочных эффектов.

  • (-) По ссылке, если функция собирается менять элементы, на которые указывает span. По значению, если не собирается.

span хранит указатель. Чтобы поменять элементы по указателю, достаточно принять его по значению. По ссылке нужно принимать, если мы хотим поменять сам указатель.
